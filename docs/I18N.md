# Internationalization (i18n)

> Complete guide to the translation system in DigiStratum applications.
> Covers static strings, dynamic content translation, and caching strategies.

---

## Overview

The i18n system uses a two-tier approach:

| Type | Description | Implementation |
|------|-------------|----------------|
| **Static strings** | UI labels, buttons, messages | Pre-translated JSON language packs |
| **Dynamic content** | User-generated data | Translate-and-cache on demand |

## Static Strings (Language Packs)

### File Structure

```
frontend/public/locales/
├── en/
│   └── translation.json
├── es/
│   └── translation.json
└── [future-lang]/
    └── translation.json
```

### Translation File Format

Each `translation.json` follows a namespaced structure:

```json
{
  "namespace": {
    "key": "Translated string",
    "keyWithVariable": "Hello, {{name}}!",
    "nested": {
      "deepKey": "Nested value"
    }
  }
}
```

### Current Namespaces

| Namespace | Purpose | Example Keys |
|-----------|---------|--------------|
| `common` | Shared UI elements | `loading`, `save`, `cancel`, `error` |
| `nav` | Navigation labels | `home`, `dashboard`, `menu` |
| `auth` | Authentication UI | `loginWith`, `unauthorized`, `loggingIn` |
| `theme` | Theme switcher | `light`, `dark`, `system` |
| `footer` | Footer content | `copyright` |

### Adding a New Language

1. Create `public/locales/{lang}/translation.json`
2. Copy structure from `en/translation.json`
3. Translate all values
4. Add language code to `supportedLngs` in `i18n/config.ts`

```typescript
// i18n/config.ts
i18n.init({
  fallbackLng: 'en',
  supportedLngs: ['en', 'es', 'fr'], // Add new language here
  // ...
});
```

### Adding New Translation Keys

1. Add key to all language files:

```json
// en/translation.json
{
  "myFeature": {
    "title": "My Feature",
    "description": "This is my new feature"
  }
}

// es/translation.json
{
  "myFeature": {
    "title": "Mi Función",
    "description": "Esta es mi nueva función"
  }
}
```

2. Use in component:

```tsx
const { t } = useTranslation();
return <h1>{t('myFeature.title')}</h1>;
```

## i18next Configuration

**Location:** `frontend/src/i18n/config.ts`

```typescript
import i18n from 'i18next';
import { initReactI18next } from 'react-i18next';
import HttpBackend from 'i18next-http-backend';
import LanguageDetector from 'i18next-browser-languagedetector';

i18n
  .use(HttpBackend)           // Load JSON files from /locales/
  .use(LanguageDetector)      // Auto-detect language
  .use(initReactI18next)      // Bind to React
  .init({
    fallbackLng: 'en',        // Default if detection fails
    supportedLngs: ['en', 'es'],
    debug: import.meta.env.DEV,
    
    interpolation: {
      escapeValue: false,     // React handles XSS
    },
    
    backend: {
      loadPath: '/locales/{{lng}}/{{ns}}.json',
    },
    
    detection: {
      order: ['localStorage', 'navigator'],
      caches: ['localStorage'],
    },
  });

export default i18n;
```

### Language Detection Order

1. **localStorage** - User's explicit choice (persisted)
2. **navigator** - Browser's language setting

### Initialization

The i18n config is imported in `main.tsx` before the app renders:

```typescript
// main.tsx
import './i18n/config';  // Initialize i18n first
import ReactDOM from 'react-dom/client';
// ...
```

## Using Translations in Components

### Basic Usage

```tsx
import { useTranslation } from 'react-i18next';

function MyComponent() {
  const { t } = useTranslation();
  
  return (
    <div>
      <h1>{t('common.loading')}</h1>
      <button>{t('common.save')}</button>
    </div>
  );
}
```

### With Variables (Interpolation)

```tsx
// Translation file
{
  "greeting": "Hello, {{name}}!",
  "items": "You have {{count}} items"
}

// Component
const { t } = useTranslation();
return (
  <>
    <p>{t('greeting', { name: user.name })}</p>
    <p>{t('items', { count: items.length })}</p>
  </>
);
```

### Pluralization

```tsx
// Translation file
{
  "item": "{{count}} item",
  "item_plural": "{{count}} items"
}

// Component - i18next auto-selects based on count
<p>{t('item', { count: 1 })}</p>  // "1 item"
<p>{t('item', { count: 5 })}</p>  // "5 items"
```

### Nested Keys

```tsx
// Translation file
{
  "errors": {
    "validation": {
      "required": "This field is required",
      "email": "Invalid email format"
    }
  }
}

// Component
<span>{t('errors.validation.required')}</span>
```

### Language Switching

```tsx
import { useTranslation } from 'react-i18next';

function LanguageSwitcher() {
  const { i18n } = useTranslation();
  
  const changeLanguage = (lng: string) => {
    i18n.changeLanguage(lng);  // Persists to localStorage
  };
  
  return (
    <select 
      value={i18n.language} 
      onChange={(e) => changeLanguage(e.target.value)}
    >
      <option value="en">English</option>
      <option value="es">Español</option>
    </select>
  );
}
```

## Language Selection Storage

### Frontend (Session/Preference)

Language preference is stored in **localStorage** via i18next-browser-languagedetector:

```typescript
// Automatic on language change
i18n.changeLanguage('es');
// → localStorage.setItem('i18nextLng', 'es')

// Manual read
const currentLang = localStorage.getItem('i18nextLng') || 'en';
```

### User Profile (Persistent)

The `User` type includes `preferredLanguage` for server-side persistence:

```typescript
// types.ts
interface User {
  id: string;
  email: string;
  name: string;
  tenants: string[];
  preferredLanguage?: string;  // ISO code: 'en', 'es', etc.
  theme?: 'light' | 'dark' | 'system';
}
```

To sync user preference on login:

```typescript
// After successful login
if (user.preferredLanguage) {
  i18n.changeLanguage(user.preferredLanguage);
}
```

## Dynamic Content Translation

For user-generated content that needs translation on-the-fly.

### Strategy: Translate-and-Cache

```
┌──────────────────────────────────────────────────────────────┐
│                    Translation Flow                           │
│                                                               │
│  1. Content arrives (e.g., user comment in Spanish)          │
│                         │                                     │
│                         ▼                                     │
│  2. Check cache: key = hash(content + targetLang)            │
│        │                                                      │
│        ├── HIT: Return cached translation                    │
│        │                                                      │
│        └── MISS: Call translation API                        │
│                         │                                     │
│                         ▼                                     │
│  3. Store in cache with TTL                                  │
│                         │                                     │
│                         ▼                                     │
│  4. Return translated content                                │
└──────────────────────────────────────────────────────────────┘
```

### Backend Translation Service (Conceptual)

```go
// internal/translation/service.go

type TranslationService struct {
    cache       Cache           // Redis or in-memory
    translator  ExternalAPI     // AWS Translate, Google, DeepL
    cacheTTL    time.Duration
}

func (s *TranslationService) Translate(
    ctx context.Context,
    text string,
    sourceLang string,
    targetLang string,
) (string, error) {
    // Same language, no translation needed
    if sourceLang == targetLang {
        return text, nil
    }
    
    // Generate cache key
    cacheKey := fmt.Sprintf("trans:%s:%s:%x", 
        sourceLang, targetLang, sha256.Sum256([]byte(text)))
    
    // Check cache
    if cached, err := s.cache.Get(ctx, cacheKey); err == nil {
        return cached, nil
    }
    
    // Call external API
    translated, err := s.translator.Translate(ctx, text, sourceLang, targetLang)
    if err != nil {
        return "", err
    }
    
    // Cache result
    s.cache.Set(ctx, cacheKey, translated, s.cacheTTL)
    
    return translated, nil
}
```

### Frontend Integration

When displaying dynamic content:

```tsx
interface TranslatableContent {
  id: string;
  text: string;
  originalLang: string;
}

function TranslatedText({ content }: { content: TranslatableContent }) {
  const { i18n } = useTranslation();
  const [translated, setTranslated] = useState<string>(content.text);
  
  useEffect(() => {
    if (content.originalLang !== i18n.language) {
      // Fetch translated version from API
      api.get<{ text: string }>(
        `/api/translate?text=${encodeURIComponent(content.text)}&to=${i18n.language}`
      ).then(res => setTranslated(res.text));
    } else {
      setTranslated(content.text);
    }
  }, [content, i18n.language]);
  
  return <span>{translated}</span>;
}
```

## Caching Strategy

### Static Translations (Language Packs)

| Layer | TTL | Strategy |
|-------|-----|----------|
| HTTP Cache | Long (1 week+) | Set `Cache-Control` headers on `/locales/` |
| i18next | Session | Loaded once, kept in memory |
| Service Worker | Indefinite | Cache-first for offline support |

**Vite configuration for caching:**

```typescript
// vite.config.ts (production build)
export default defineConfig({
  build: {
    rollupOptions: {
      output: {
        // Hash locale files for cache busting
        assetFileNames: (assetInfo) => {
          if (assetInfo.name?.endsWith('.json')) {
            return 'locales/[name]-[hash][extname]';
          }
          return 'assets/[name]-[hash][extname]';
        },
      },
    },
  },
});
```

### Dynamic Translations (Backend Cache)

| Layer | TTL | Strategy |
|-------|-----|----------|
| Redis/DynamoDB | 24-48 hours | Content hash as key |
| Memory (LRU) | 5 minutes | Hot translation cache |

**Cache key structure:**

```
trans:{sourceLang}:{targetLang}:{contentHash}

Example:
trans:es:en:a1b2c3d4e5f6...
```

### Cache Invalidation

- **Static strings:** Deploy new language pack files (hash changes → new URL)
- **Dynamic content:** TTL expiration; no manual invalidation needed
- **User preference change:** No cache impact (different lang = different key)

## Best Practices

### 1. Keep Translation Keys Semantic

```json
// ✅ Good - semantic keys
{
  "auth": {
    "loginButton": "Sign In",
    "logoutButton": "Sign Out"
  }
}

// ❌ Bad - generic or positional keys
{
  "button1": "Sign In",
  "text_at_top": "Welcome"
}
```

### 2. Avoid Concatenation

```tsx
// ❌ Bad - breaks in other languages
t('greeting') + ' ' + userName + '!'

// ✅ Good - use interpolation
t('greeting', { name: userName })
// "Hello, {{name}}!" → "Hello, John!"
```

### 3. Handle Missing Translations

```typescript
// i18n/config.ts
i18n.init({
  // ...
  saveMissing: import.meta.env.DEV,  // Log missing keys in dev
  missingKeyHandler: (lng, ns, key) => {
    console.warn(`Missing translation: ${lng}/${ns}/${key}`);
  },
});
```

### 4. Test All Languages

```tsx
// In tests, override language
beforeEach(() => {
  i18n.changeLanguage('es');
});

it('renders Spanish text', () => {
  render(<MyComponent />);
  expect(screen.getByText('Guardar')).toBeInTheDocument();
});
```

### 5. Document Translation Context

```json
{
  "save": "Save",
  "_save_context": "Button label for saving forms (verb, imperative)",
  
  "file": "File",
  "_file_context": "Noun, as in 'a file' (document)"
}
```

*(Context keys starting with `_` are ignored by i18next but help translators)*

## Adding a New Supported Language

### Checklist

1. **Create translation file:**
   ```bash
   cp public/locales/en/translation.json public/locales/{lang}/translation.json
   ```

2. **Translate all strings** in the new file

3. **Update i18n config:**
   ```typescript
   supportedLngs: ['en', 'es', 'fr'], // Add new code
   ```

4. **Add to language switcher UI** (if applicable)

5. **Test the new language:**
   ```bash
   # Force language for testing
   localStorage.setItem('i18nextLng', 'fr');
   location.reload();
   ```

6. **Update documentation** (this file, README if public-facing)

## Troubleshooting

### Translation Not Appearing

1. Check browser console for missing key warnings
2. Verify file exists: `/locales/{lang}/translation.json`
3. Check network tab - is the JSON loading?
4. Verify key path matches exactly (case-sensitive)

### Wrong Language Loading

1. Check `localStorage.getItem('i18nextLng')`
2. Clear localStorage and reload
3. Check browser language settings
4. Verify `supportedLngs` includes the language

### Interpolation Not Working

```tsx
// ❌ Wrong - missing variable in translation
t('greeting')  // "Hello, {{name}}!" → "Hello, {{name}}!"

// ✅ Correct
t('greeting', { name: 'John' })  // "Hello, John!"
```

### Performance Issues

- Large translation files: Split into namespaces
- Too many dynamic translations: Review cache TTL
- Slow language switch: Pre-load common languages
